# 存储占用

存储占用页面用于查看 Omnibot 当前的本地空间使用情况，并对可回收的数据执行清理。它不是一个只显示数字的统计页，而是把“分析占用来源”和“按风险做清理”放在了一起，方便你在空间紧张、模型导入失败、缓存堆积或长期使用之后快速定位问题。

## 页面定位

这个页面主要解决两类问题：

- 当前到底是哪些内容在占空间
- 哪些内容可以安全清理，哪些需要谨慎处理


## 核心能力

## 1. 存储概览

页面顶部会先给出一组概览信息，用来快速判断当前空间状态。

这里包括：

- 当前已占用的总量
- 可以回收的空间规模
- 最近分析结果对应的状态变化


## 2. 清理建议

当系统判断是、当前占用已经具备明显的可回收空间时，会给出一组推荐的清理策略。例如：

- 优先清理低风险缓存
- 先做保守清理，再决定是否继续
- 针对明显占用异常的内容执行集中回收

## 3. 分类占用

页面下半部分会把占用拆到不同分类，方便你判断空间究竟花在了哪里。

这类分类通常用于区分不同来源的数据，例如：

- 缓存类内容
- 下载或导入的资源
- 工作区相关文件
- 附件、生成物或历史执行结果
- 其他应用内部持久化数据

通过饼状图，可以清晰看出各部分空间占用比例，之后再决定是否清理，而不是盲目一键删除。

![存储页概览](/assets/storage1.jpg)


## 4. 按分类清理

对于支持清理的分类，页面会直接提供清理入口。

这样做可以直观清理掉自己想清理的内容，安全性高。


![分类清理](/assets/storage2.jpg)


## 使用建议

如果你是第一次清理，建议按下面的顺序处理：

1. 先看概览，确认是否真的需要清理
2. 再看分类占用，找到主要来源
3. 优先清理低风险的内容
4. 如果只是临时腾空间，再考虑执行推荐策略
5. 清理后重新分析一次，确认结果是否符合预期


## 注意事项

- 不是所有分类都适合做“部分保留清理”。某些高风险分类可能只支持整体处理。
- 清理缓存通常影响较小，但清理历史产物、附件或工作区相关文件时，建议先确认这些内容是否仍需要保留。
- 如果你主要的问题来自本地模型占用，建议结合 `本地模型` 页面一起处理，因为模型删除、导入和服务状态管理不完全在这一页完成。
- 页面里的“可清理空间”更适合用来帮助决策，不一定等于最终能完全释放的系统空间。
